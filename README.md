# Cellar

> An ML-powered intelligence layer for your Obsidian vault.

Cellar ingests your markdown notes, embeds them into vector space, clusters them by semantic similarity, and surfaces the hidden structure of your knowledge — what topics you actually think about, what concepts are silently related but never linked, and where your blind spots are.

**Obsidian plugin: TBD.** If this becomes a plugin, the first priority will be incremental updates — right now the pipeline re-embeds and re-clusters the entire vault from scratch on every run rather than processing only changed files.

---

## What It Does

- **Semantic Embeddings** — Every note is embedded into a 768-dimensional vector using `all-mpnet-base-v2`, capturing meaning rather than just keyword overlap.
- **Knowledge Map** — UMAP reduces those embeddings to 2D so the structure of your vault can actually be plotted and seen, not just computed.
- **Topic Clustering** — HDBSCAN groups semantically similar notes into clusters without requiring you to specify how many topics exist ahead of time.
- **Topic Labeling** — Each cluster gets a short, human-readable label generated by Claude (Haiku), so "Cluster 17" becomes "Front End Styling & Layout."
- **Link Recommendations** — For every note, Cellar finds the most semantically similar notes that have no existing wikilink, and surfaces them as suggested connections.
- **Vault Write-Back** — Recommendations are appended directly into each note under a `##### Cellar Recommendations` section, so you can review and manually accept/reject them inside Obsidian.
- **Orphan Detection** — Notes that HDBSCAN can't confidently group with anything else are flagged as noise — these are your semantically isolated notes.

On my own 231-note vault, this pipeline identified 31 distinct topic clusters and 32 orphaned notes, and generated link recommendations across the entire vault.

---

## Tech Stack

- **sentence-transformers** (`all-mpnet-base-v2`) — note embeddings. Chosen over `all-MiniLM-L6-v2` for meaningfully better semantic accuracy, at an embedding-time cost that's still acceptable for a vault of a few hundred notes.
- **UMAP** — dimensionality reduction from 768D embeddings down to 2D coordinates for visualization, using cosine distance to match how the embeddings were generated.
- **HDBSCAN** — density-based clustering on the UMAP output. Chosen over k-means because the number of topic clusters in a personal vault isn't known ahead of time, and HDBSCAN naturally produces a noise/outlier category instead of forcing every note into a group.
- **Anthropic API (Claude Haiku)** — generates short topic labels for each cluster from its note titles.
- **matplotlib** — renders the final knowledge map, color-coded by cluster and annotated with LLM-generated labels.
- **NumPy** — caching and array operations throughout the pipeline.

## Setup

### 1. Environment

```bash
conda create -n cellar python=3.11
conda activate cellar
pip install sentence-transformers umap-learn hdbscan matplotlib anthropic numpy
```

### 2. Anthropic API key

Cluster labeling calls the Claude API. Get a key from [console.anthropic.com](https://console.anthropic.com), then set it as an environment variable so it's never hardcoded:

```bash
export ANTHROPIC_API_KEY="sk-ant-your-key-here"
```

Add that line to your `~/.zshrc` (or `~/.bashrc`) so it persists across terminal sessions.

### 3. Run the pipeline

All commands are run from the project root. Run them in this order — each step depends on the cached output of the one before it.

```bash
# 1. Parse your vault and generate embeddings + similarity matrix
python backend/embeddings/embedder.py --vault-path "/path/to/your/Obsidian/Vault"

# 2. Reduce embeddings to 2D for visualization
python backend/clustering/umap_reducer.py

# 3. Cluster notes by topic
python backend/clustering/clusterer.py

# 4. Generate link recommendations
python backend/clustering/recommender.py

# 5. Generate LLM labels for each cluster
python backend/clustering/labeler.py

# 6. (Optional, writes to your real vault) Append recommendations into your notes
python backend/clustering/note_writer.py

# 7. Generate the knowledge map visualization
python backend/visualization/plot_clusters.py
```

**The only thing you need to change is `--vault-path` in step 1** — point it at your own Obsidian vault's root folder. Everything downstream reads from cached files in `data/vault_cache/` and requires no further configuration.

> ⚠️ Step 6 modifies your actual vault files by appending a recommendations section to the bottom of each note. It's idempotent (running it twice won't duplicate the section), but back up your vault or test on a copy first if you want to be careful.

## Project Structure

```
cellar/
├── backend/
│   ├── ingestion/vault_parser.py     — walks the vault, strips frontmatter, extracts wikilinks
│   ├── embeddings/embedder.py        — embeds notes, caches embeddings + similarity matrix
│   ├── clustering/
│   │   ├── umap_reducer.py           — 768D → 2D for visualization
│   │   ├── clusterer.py              — HDBSCAN topic clustering
│   │   ├── recommender.py            — link recommendation engine
│   │   ├── labeler.py                — LLM cluster labeling via Claude
│   │   └── note_writer.py            — writes recommendations back into vault notes
│   └── visualization/plot_clusters.py — generates the knowledge map
├── data/vault_cache/                 — generated cache (gitignored, except this file is regenerated fresh each run)
└── TUNING_LOG.md                     — HDBSCAN parameter tuning experiments and reasoning
```

## Known Limitations

- **No incremental updates.** Adding a single note to your vault requires re-running the full pipeline from `embedder.py` onward. This is the first thing an Obsidian plugin version would need to solve.
- **Recommendation count isn't guaranteed.** The top-k similarity filter can return fewer than `k` recommendations if some candidates get filtered out (already-linked, or the note itself) — it doesn't currently backfill to guarantee exactly 5.
- **Hardcoded module path resolution.** Scripts use `sys.path.append` to resolve internal imports, which assumes a specific clone location rather than being a properly packaged Python module.

## Tuning Notes

HDBSCAN's `min_cluster_size` and `min_samples` were tuned across multiple runs by inspecting the actual notes inside each resulting cluster, not just by cluster count. Full reasoning and intermediate results are in [`TUNING_LOG.md`](./TUNING_LOG.md). Final parameters: `min_cluster_size=3, min_samples=2`.
